# 工程与构建规范

更新时间：2026-05-08

## 基本原则

- 先读项目文件和现有代码风格，再改代码。
- 不回退用户或其他协作者的改动。
- 构建、测试、复制产物要记录清楚路径和结果。
- 修改已有文件用 `search_replace`；新建文件用 `create_file`；整行替换禁用 echo/sed。

## 构建路径规约（铁律）

- **OutputPath**（MOD 的 DLL 编译输出）：统一指向 `c:\Users\Administrator\Desktop\MOD\PEAK\测试环境\BepInEx\plugins\`。
- **HintPath**（引用路径）：
  - BepInEx 核心、0Harmony、第三方 MOD（`BlackPeakRemix`、`Thanks.ShootZombies` 等） → 指向 `测试环境\BepInEx\`。
  - 游戏本体 DLL（`Assembly-CSharp.dll`、`UnityEngine.*`、`Photon*`、`pworld` 等） → 保持 Steam 原路径 `C:\SteamLibrary\steamapps\common\PEAK\PEAK_Data\Managed\`，不要改。
- **OutputPath 不碰 Steam 目录**：开发期的 DLL 只进测试环境，正式发布再手动打包到 `发行/<版本号>/`。

## BepInEx / Unity MOD

- 优先保持 MOD 独立，不把原 MOD DLL 当补丁包裹，除非用户明确要求。
- 反编译代码只能作为参考，最终代码要符合当前工程结构和命名风格。
- 对 Unity 对象、PhotonView、RPC、场景对象生成和销毁要保守处理，避免破坏游戏运行状态。
- **热更限制**：BepInEx 插件 DLL 被游戏进程锁定，更新前必须先退出游戏；`run_in_terminal` 跑的 `dotnet build` 如果提示 DLL 写入失败，十有八九是游戏没关。

## 项目重构参照标准

以 `Lantern&ShootZombies&Night` 为标杆重构其他 MOD：

- **SDK 风格 csproj**：`<Project Sdk="Microsoft.NET.Sdk">`，净化引用声明。
- **Plugin.cs 入口拆分**：主 Plugin 只放 `[BepInPlugin]` 与生命周期，业务逻辑拆到 `Plugin.Processing.cs` 或独立 helper。
- **Patches 目录隔离**：所有 Harmony 补丁独立文件，一个补丁一个类。
- **私有成员访问**：统一用 `AccessTools + Traverse` 方案（HarmonyX API），别直接反射。

## 构建输出记录

- 每个 MOD 的构建命令和输出路径写在 `mods/<ModName>/FILES.md`。
- 构建完成后必须说明 DLL 是否已同步到测试环境。
- 如果需要用户手动替换 DLL，要明确告诉用户替换哪个文件。
