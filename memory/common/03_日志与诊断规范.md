# 日志与诊断规范

更新时间：2026-05-08

## 日志目标

- 事后能从日志判断问题发生在哪个阶段。
- 关键路径必须能看到**输入、输出、耗时、异常和跳过原因**。
- 日志不能只写"失败"，必须带上下文。

## 日志分级约定

| 级别 | 用途 |
|------|------|
| `LogDebug` | 详细链路、诊断帮助，默认关闭。功能验证完后**禁止**仍打 Info 级。 |
| `LogInfo` | 正常业务路径的关键节点，例如配置加载、版本输出、功能启用结果。 |
| `LogWarning` | 非致命异常、回退路径、反射找不到目标等。 |
| `LogError` | 功能中断、RPC 校验失败、炸房事件、异常堆栈。 |

**铁律**：日常调试代码在上传/发布前必须降级为 `LogDebug` 或移除，不允许提交大量 Info 噪音。

## 推荐记录

- MOD 初始化、版本号、配置加载结果。
- 扫描到的场景、地图、对象、生成组、动态名称。
- 运行时生成、清理、刷新、导入、导出、预览相机状态。
- 多人同步相关的**房主、客户端、PhotonView、RPC、种子和属性同步结果**。
- 异常堆栈和可恢复策略。

## 结构化日志与 FieldProbe（WhySoLaggy 体系）

WhySoLaggy 提供一套结构化诊断基建，其他 MOD 需要深度诊断时优先复用：

- `StructuredLogger`：键值对日志，便于日志后处理工具抓取。
- `FieldProbe`：运行时字段值探针，通过 DSL 配置动态采样。
- `RpcMonitor`：RPC 来源、频率、发送者统计，用于炸房溯源。
- `PatchProfiler` / `PluginProfiler`：Harmony 补丁与插件耗时归因。
- `HarmonyScanner`：扫全体 Harmony 补丁注册情况。

### FieldProbe DSL 语法

配置文件位置：`测试环境\BepInEx\config\WhySoLaggy.fieldprobe.json`（实际在 Steam 游戏目录的 BepInEx/config 下，不在开发目录）。

DSL 基本格式：

```
Type.Method >> 字段表达式1, 字段表达式2, ...
```

- `Type.Method`：要挂探针的目标方法（完整类名）。
- 字段表达式支持属性链、索引器、静态字段。
- 默认**关闭**，需在 JSON 里显式启用 `enabled: true`。

### JSON 配置文件注意

- 标准 JSON **禁用 `//` 和 `/* */` 注释**。
- 说明信息必须用合法 JSON 字段承载：
  - `_doc`：全局/顶层说明。
  - `note`：单条规则说明。

## 诊断目录

- 每个 MOD 的诊断目录写在 `mods/<ModName>/FILES.md`。
- 用户说"日志已更新"时，**优先读取对应诊断目录的新文件**。
- 诊断结论要同步到对应 MOD 的 `RECENT.md` 或根目录 `TODO.md`。
