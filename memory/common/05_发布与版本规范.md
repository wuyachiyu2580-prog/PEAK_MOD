# 发布与版本规范

更新时间：2026-05-08

## 版本号

- 用户明确指定版本号时，以用户要求为准。
- 未发布前**不要擅自滚动版本号**。
- 版本号变更要同步：`.csproj`、`manifest.json`、`README.md` 头部、`CHANGELOG.md`、memory 里对应 MOD 的 README/FILES。
- 代码里任何含版本号的注释都要一起更新，别留旧版本号漂字符串。

## 三文件协同（发行包必带）

每次发布到发行目录 `MOD开发/<MOD>/发行/<版本号>/` 时，必须产出三件套：

1. **README.md**：面向玩家的白话中文，含安装步骤、功能说明、配置表、注意事项。
2. **CHANGELOG.md**：本次版本做了什么，用玩家能懂的语言。
3. **manifest.json**：Thunderstore 规范，声明依赖。

### manifest.json 撰写规范

- **description 字段 ≤ 135 字符**，只留最核心玩法说明（补灯方式、终极技能效果、语言支持、依赖版本）。细节搬到 README。
- **dependencies**：只声明确实必须的，例如 `BepInEx-BepInExPack_PEAK`。别把开发期用的引用当运行期依赖。
- **JSON 禁注释**：要加说明用 `_doc`（全局）或 `note`（单条），不要写 `//` 或 `/* */`。

## README 四维结构

对外 README 按这四块写，接手者能快速抓到重点：

1. **是什么**：一句话功能简介。
2. **怎么用**：安装步骤 + 按键/入口。
3. **怎么调**：配置项说明（含默认值与效果）。
4. **已知问题 / 注意事项**：别掩盖 bug，老实说。

## 文档三阶段维护

对外文档（README / CHANGELOG）更新走三阶段：

1. **修复阶段**：先写清楚"改了啥"（技术实现）。
2. **易读化**：把技术词翻译成玩家能懂的白话中文。
3. **双语/美化**（可选）：配图、emoji、分区标题，提升可读性。

> 用户强调过：对外文档别用官方口吻，**像跟玩家聊天一样写**。

## 发布前检查清单

- [ ] 构建通过，DLL 已落到测试环境。
- [ ] 关键功能在测试环境跑过（单人 + 多人）。
- [ ] 配置、依赖、输出路径、诊断目录文档明确。
- [ ] 版本号全链路一致（csproj / manifest / README 顶部 / CHANGELOG）。
- [ ] 日志级别降到位，无大量 Info 噪音。
- [ ] 已知问题写入 `TODO.md` 或发布说明。
- [ ] 发行目录 `发行/<版本号>/` 下三文件齐全。
