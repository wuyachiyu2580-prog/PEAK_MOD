# 联机与同步规范

更新时间：2026-05-08

## 基本原则

- 房主（Master）和客户端（Client）的职责要明确。
- 配置同步、运行时状态同步、HUD 显示同步**不要混在一起**。
- 多人中途加入必须考虑**已有状态回放或重新同步**。

## 主客机权限分工

| 类型 | 归属 | 同步手段 |
|------|------|---------|
| 配置参数（灯笼燃料上限、抗寒倍率、HUD 预设等） | 主机决定 | 加入房间时主机→客户端一次性下发 |
| 运行时状态（灯笼点亮、温度、体力） | 对象持有方 | PhotonView + RPC |
| 本地 HUD 显示 | 每个客户端自己 | 只读本地状态，不发 RPC |

**铁律**：客户端不能篡改配置参数；主机也不能越权改客户端 HUD。

## Photon 相关

- 修改 PhotonView、RPC、房间属性和玩家属性前，先查当前 MOD 是否已有同步约定。
- 不要频繁创建或销毁 PhotonView，先评估性能和 ID 冲突风险。
- 种子、地图生成参数和运行时对象变更需要**明确同步来源**。
- RPC 命名遵循 `MOD名_动词_对象` 惯例，方便 `RpcMonitor` 归因。

## RPC 校验与防刷（炸房溯源）

用户强调过的铁律——**所有对外 RPC 必须带校验**，防止被恶意客户端刷爆：

- **来源校验**：`sender == master` 才接受 broadcast 类 RPC。
- **频率限流**：单玩家每秒 RPC 数超阈值记入 `NetworkAbuseDetector`。
- **载荷校验**：数值类参数必须校验范围，字符串类校验长度与白名单字符。
- **拒绝策略**：校验失败直接丢弃并打 `LogWarning`，不触发业务逻辑。

WhySoLaggy 的 `RpcMonitor` 负责记录：
- 每个 RPC 的调用次数、发送者、参数摘要。
- 异常高频 RPC 触发 `NetworkAbuseDetector` 告警。
- 用于房间"炸房"事件的事后溯源。

## 中途加入同步

- 新玩家入房时，主机必须**广播当前配置快照**，不要等对方触发事件。
- 运行时生成的对象需要能重放，或重新生成时带同一个种子。
- HUD 相关状态（谁点了灯、谁死了）必须在入房时重建，不靠事件流补齐。

## 排查方向

- 主客机状态是否一致。
- 房间属性是否在加入中途回放。
- RPC 是否只发给当前在线玩家。
- 生成对象是否依赖本地随机状态而未同步种子。
- 配置分区是否覆盖所有参数（有新参数忘了同步就会出现"主机正常客户端异常"）。
